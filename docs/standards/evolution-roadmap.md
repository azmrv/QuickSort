## 1. ЦЕЛЬ ПЛАНА

Обеспечить стабильность, поддерживаемость и эволюционную гибкость проекта QuickSort путём системного аудита кода, исправления ошибок, рефакторинга и внедрения улучшений архитектуры. Работа выполняется с соблюдением строгих правил внесения изменений и документирования, чтобы сохранить целостность проекта и избежать регрессий.

### Фаза 1. Аудит кода (поиск ошибок, проблем и нарушений)

**Цель:** Выявить все критические и некритические проблемы в коде, архитектуре, безопасности и производительности.

#### 1.1. Анализ архитектуры и зависимостей

- **Проверить соблюдение Dependency Rule** (Domain не зависит от Application и Infrastructure). Использовать скрипт `docs/standards/architecture_guard.py`.
- **Проверить согласованность слоёв** — соответствуют ли реальные импорты описанным портам.
- **Выявить циклы** в зависимостях между крейтами (cargo tree).
- **Проверить версии зависимостей** — нет ли устаревших или уязвимых крейтов.

#### 1.2. Анализ кода на предмет ошибок и узких мест

| Область | Инструменты | Что искать |
|---------|-------------|------------|
| **Domain** | `cargo clippy`, ручной просмотр | Инварианты, нарушения иммутабельности, ошибки в логике Operation.reverse() |
| **Application** | `cargo test -- --nocapture` | Неполная обработка ошибок, отсутствие таймаутов, утечки контекста |
| **Infrastructure** | `cargo audit`, `cargo bench` | Небезопасные операции с fs, отсутствие fallback при ошибках JSON |
| **Tauri/React** | `npm audit`, React DevTools | Утечки памяти, необработанные промисы, неоптимальные ререндеры |
| **IPC (Named Pipe)** | Ручной анализ | Обработка разрыва соединения, таймауты, сериализация/десериализация |

#### 1.3. Анализ безопасности

- **Проверить валидацию входных данных** на всех границах (Tauri-команды, IPC, CLI).
- **Проверить обработку путей** — нет ли возможности выхода за пределы разрешённых директорий (path traversal).
- **Проверить логирование** — не попадают ли в логи чувствительные данные (пароли, пути пользователя).
- **Проверить работу с реестром** — только HKEY_CURRENT_USER.

#### 1.4. Анализ производительности

- **Замерить время** открытия окна выбора, перемещения файлов, загрузки списка папок.
- **Проверить использование памяти** — нет ли утечек в долгоживущих структурах.
- **Проверить конкурентность** — корректно ли используются async/await, нет ли блокирующих операций в асинхронном коде.

**Результат Фазы 1:** Список проблем (багов, нарушений стандартов, рисков) с приоритетами (Critical, High, Medium, Low). Каждая проблема документируется в `docs/workspace/audit-issues.md`.

### Фаза 2. Приоритизация и планирование рефакторинга

**Цель:** Определить очерёдность исправлений и разбить работу на управляемые задачи.

**Действия:**
1. Классифицировать проблемы по критичности:
   - **Critical** — падения, потеря данных, уязвимости → исправлять немедленно.
   - **High** — серьёзные нарушения логики, большие утечки памяти → исправлять в первую очередь.

   - **Low** — улучшения читаемости, опциональные оптимизации → при возможности.

2. Сгруппировать проблемы по слоям (Domain, Application, Infrastructure, Adapters) и модулям.
3. Создать задачи в `docs/workspace/tasks/` с привязкой к TASK-XXX.
4. Определить порядок рефакторинга (снизу вверх: Domain → Application → Infrastructure → Adapters).

**Результат Фазы 2:** План рефакторинга с задачами, приоритетами и сроками (примерно).

### Фаза 3. Рефакторинг (поэтапное внесение изменений)

**Цель:** Исправить выявленные проблемы, улучшить код, не нарушая работоспособность.

**Ключевые правила выполнения (дополнены в `refactor_rules.md`):**
- **Не удалять файлы и содержимое файлов целиком.** Вместо этого комментировать старые строки, а новые добавлять с комментарием.
- **При добавлении или замене кода писать подробные комментарии на английском языке**, объясняющие «что» и «почему».
- **При работе с новыми библиотеками или API сначала искать примеры использования в Google.com** и только после понимания применять их.
- **Все TODO фиксировать в коде** в виде комментариев `// TODO: <описание>`, и одновременно заносить в документацию, изменяя соответствующие разделы (структуру файлов, имена и т.п.).

#### 3.1. Рефакторинг Domain Layer (`quicksort-domain`)

- Исправить инварианты (например, проверка пустых путей).
- Добавить недостающие доменные события.
- Улучшить иммутабельность структур.
- Пересмотреть иерархию ошибок (использовать `thiserror`).

#### 3.2. Рефакторинг Application Layer (`quicksort-application`)

- Добавить контексты и таймауты ко всем Use Case.
- Улучшить обработку ошибок (возвращать конкретные типы, а не `anyhow`).
- Убедиться, что фасад (`Facade`) является единственной точкой входа.
- Проверить корректность DTO и маппинга.

#### 3.3. Рефакторинг Infrastructure Layer (`quicksort-infrastructure`)

- Переписать репозитории с использованием асинхронных операций ввода-вывода.
- Добавить кеширование для часто читаемых данных.
- Улучшить обработку ошибок файловой системы (retry, fallback).
- Пересмотреть стратегию разрешения конфликтов (`ConflictResolver`).

#### 3.4. Рефакторинг Adapters (Tauri и React)

- Удалить устаревший код из `src-tauri/src/folder/` (заменить на вызовы фасада).
- Унифицировать Tauri-команды: каждая команда должна вызывать только фасад.
- В React-фронтенде: вынести логику работы с Tauri в отдельные хуки, улучшить обработку ошибок и состояния загрузки.
- Пересмотреть дизайн компонентов на соответствие Ant Design и принципам доступности.

#### 3.5. Рефакторинг IPC и Context Menu DLL

- Улучшить обработку разрыва соединения (переподключение, таймауты).
- Добавить верификацию данных от клиента (валидация путей, типов операций).
- Унифицировать протокол (использовать контракт из `quicksort-ipc-contract`).

## 2. Rust Style

Основа — стандартный стиль `rustfmt` с дополнительными правилами:

### 2.1. Именование

| Категория | Стиль | Пример |
|-----------|-------|--------|
| Типы (struct, enum, trait) | `PascalCase` | `Folder`, `Operation` |
| Функции, методы, переменные | `snake_case` | `move_files()`, `folder_id` |
| Константы | `SCREAMING_SNAKE_CASE` | `MAX_RETRIES` |
| Модули (файлы) | `snake_case` | `operation_repository.rs` |

### 2.2. Структура файла

```rust
// 1. Импорты (группировать)
use std::path::PathBuf;
use serde::{Deserialize, Serialize};

// 2. Модульные объявления (если есть)
mod tests;

// 3. Константы
const DEFAULT_LIMIT: usize = 50;

// 4. Структуры и реализации
pub struct Folder { ... }
impl Folder { ... }

// 5. Трейты и реализации
pub trait FolderRepository { ... }

// 6. Функции (публичные, затем приватные)
pub fn create_folder() { ... }
fn helper() { ... }

// 7. Тесты (в конце, под #[cfg(test)])
```

### 2.3. Обработка ошибок

- Использовать `thiserror` для доменных ошибок (в Domain).
- В Application возвращать конкретные ошибки, а не `anyhow`.
- Логировать ошибки через `tracing::error!` с контекстом.

```rust
#[derive(Debug, thiserror::Error)]
pub enum FolderError {
    #[error("папка уже существует: {path}")]
    AlreadyExists { path: String },
    #[error("папка не найдена: {id}")]
    NotFound { id: FolderId },
}

fn get_folder(id: FolderId) -> Result<Folder, FolderError> { ... }
```

## 3. TypeScript / React Style

interface FolderListProps {

export const FolderList: React.FC<FolderListProps> = ({ folders, onSelect }) => {

    setSelectedId(id);
    onSelect(id);
  };

    </div>
  );

### 3.2. `docs/standards/refactor_rules.md` — Правила внесения изменений

```markdown
# Refactoring Rules — QuickSort Project

**Версия:** 1.0  
**Дата:** 2026-07-11  
**Применяется к:** Все изменения в коде, документации и архитектуре.

---

## 1. Общие принципы

1. **Безопасность прежде всего** — каждое изменение должно быть проверяемым и откатываемым.
2. **Прозрачность** — все изменения документируются в коде и в документации.
3. **Постепенность** — рефакторинг выполняется маленькими шагами, каждый из которых завершается тестированием.
4. **Сохранение совместимости** — внешние API (Tauri-команды, IPC-протокол) не ломаются без предварительного уведомления и миграции.

---

## 2. Процесс внесения изменений

### 2.1. Перед изменением

1. Создать задачу в `docs/workspace/tasks/` (или найти существующую).
2. Изучить связанные ADR и стандарты.
3. **Поискать примеры использования** новых библиотек или API в Google.com (или через поисковик) и только после понимания применять их.
4. Написать план изменений (кратко).

### 2.2. Во время изменения

1. **Не удалять файлы и содержимое файлов целиком.**
   - Вместо удаления — закомментировать старые строки (с `// OLD:`) и добавить новые.
   - Удалять файлы только если они полностью устарели и не используются (согласовать с командой).

2. **Комментировать каждое существенное изменение:**
   ```rust
   // OLD: неверная проверка
   // if path.exists() { ... }
   // NEW: корректная проверка с контекстом
   if !tokio::fs::try_exists(&path).await? {
       return Err(Error::NotFound);
   }
   ```

3. **Добавлять комментарии на английском языке** для новых блоков кода, объясняя причину изменения:
   ```rust
   // Retry up to 3 times with exponential backoff to handle transient file system errors.
   for attempt in 0..3 {
       if let Ok(result) = fs::rename(&src, &dst).await {
           return Ok(result);
       }
       tokio::time::sleep(Duration::from_millis(100 * 2_u64.pow(attempt))).await;
   }
   ```

4. **Фиксировать TODO** в коде:
   ```rust
   // TODO: TASK-015 — replace with async lock to avoid deadlocks
   let _lock = mutex.lock().await;
   ```

### 2.3. После изменения

1. Запустить тесты (локально).
2. Обновить документацию:
   - Если изменились имена файлов или структура папок — обновить `SPECIFICATION.md` и `AGENTS.md`.
   - Если изменился публичный API — обновить соответствующие `docs/architecture/*.md`.
   - Если добавилось новое архитектурное решение — создать новый ADR.
3. Внести запись в `CHANGELOG.md` с номером версии (по схеме `1.0.0.0-XXXXX`).

## 3. Правила для отдельных слоёв

### 3.1. Domain

- Не добавлять зависимости от внешних крейтов (кроме `serde` для сериализации).
- Все изменения должны быть обратимыми (reverse operation для Undo).

### 3.2. Application

- Использовать `tokio` для асинхронности, добавлять таймауты к каждому Use Case.
- Не использовать `unwrap()` или `expect()` — только `?` и явная обработка ошибок.

### 3.3. Infrastructure

- Операции с файловой системой должны быть атомарными (использовать временные файлы, затем переименовывать).
- Кешировать результаты `ConfigurationRepository` на 5 секунд.

### 3.4. Adapters (Tauri / React)

- Tauri-команды должны быть тонкими обёртками над фасадом.
- React-компоненты — использовать React.memo для предотвращения лишних ререндеров.

---

## 4. Исключения и отступления

- **Критические исправления** (security, data loss) могут быть выполнены вне очереди, но обязательно документируются.
- **Экспериментальные изменения** (новые фичи) выполняются в отдельных ветках и не сливаются в main до полного тестирования.
