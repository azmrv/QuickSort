# План реализации QuickSort v2.0 (По состоянию на 2026-07-11)

Данный документ описывает поэтапный план реализации ключевых функций и архитектурных улучшений, основанный на спецификациях `SPECIFICATION.md` и ADR. Цель — обеспечить стабильную, расширяемую и тестируемую платформу для выполнения файловых операций.

## 🎯 Общая цель
Завершить разработку ядра бизнес-логики (Domain/Application) для поддержки всех основных операций: Move, Copy, Delete, Rename, а также реализовать механизм отката (`Undo`) для каждой из них.

## 🛠️ Этап 1: Уточнение и унификация DTO (Выполнено)
*   **Действие:** Обновление `OperationCommand` в `quicksort-application/src/dtos/operation_command.rs`.
    *   Добавлены поля для явной поддержки операций **Delete** и **Rename**.
    *   Внедрена унифицированная обработка целевых путей (`target_paths`) для всех операций, где это необходимо (особенно Rename).
*   **Статус:** ✅ Завершено.

## 🛠️ Этап 2: Реализация логики операций в Application Layer
Основная задача — обновить `ExecuteOperationUseCase` и его зависимые компоненты для обработки новых типов операций.

### 2.1. Обновление Use Case `ExecuteOperationUseCase`
*   **Цель:** Расширить метод `execute_operation(command: OperationCommand)` для поддержки Delete, Rename и Copy.
*   **Действия:**
    1.  **Delete:** Реализовать логику удаления файлов/папок с учетом политики конфликтов (если папка содержит файлы). Добавить в процесс генерацию события `OperationFailed` или `OperationCompleted`.
    2.  **Rename:** Использовать `target_paths` для определения нового имени и пути. Логика должна быть атомарной, используя возможности файловой системы (`std::fs::rename`).
    3.  **Copy:** Реализовать логику копирования (по требованию). Это потребует дополнительной обработки потоков данных и учета размера файлов.
*   **Зависимости:** Требуется тесное взаимодействие с `ConflictResolver` для каждой операции, чтобы обеспечить атомарность и корректное разрешение конфликтов имен.

### 2.2. Обновление Domain Entities (`quicksort-domain/src/entities/mod.rs`)
*   **Цель:** Обеспечить, что сущности `Operation` могут хранить метаданные для всех типов операций.
*   **Действия:** Проверить и, при необходимости, обновить поля в структуре `Operation` (например, добавить поле для хранения оригинального имени/пути при Rename).

### 2.3. Обновление Логики Отката (`UndoOperationUseCase`)
*   **Цель:** Обеспечить возможность отката всех операций (Move, Copy, Delete, Rename).
*   **Действия:**
    1.  Для каждой операции необходимо определить **обратную операцию**.
        *   `Undo(Move)`: Выполнить `Move` обратно из цели в источник.
        *   `Undo(Rename)`: Переименовать обратно (из нового имени в старое).
        *   `Undo(Delete)`: Восстановить удаленный объект (требует временного хранилища или корзины).
    2.  Обновить `OperationState::Undone` и логику `mark_undone` для учета типа операции, которая была отменена.

## 🛠️ Этап 3: Инфраструктура и Тестирование (Тесты)
*   **Цель:** Обеспечить покрытие всех новых сценариев тестами.
*   **Действия:**
    1.  Создать/обновить тестовые сценарии в `crates/quicksort-application/tests/scenarios/execute_operation/*`.
        *   `delete.rs`: Тестирование удаления файлов и папок, включая обработку пустых и непустых директорий.
        *   `rename.rs`: Тестирование переименования с различными сценариями конфликтов.
        *   `copy.rs`: Тестирование копирования больших объемов данных.
    2.  Обновить `ConflictResolver` для учета специфики Delete/Rename (например, при удалении папки нужно ли проверять содержимое).

## 🚀 Этап 4: Интеграция и Финализация
*   **Действие:** Обновление фасада (`Facade`) и команд Tauri.
    1.  Убедиться, что `Facade` вызывает обновленный `ExecuteOperationUseCase` с корректным DTO для всех типов операций.
    2.  Проверить логирование: все операции должны генерировать соответствующие события (`OperationCompleted`, `OperationFailed`).
