## Спецификация проекта QuickSort (актуализированная)

**Версия:** 2.0  
**Дата:** 2026-07-11  
**Статус:** Активна  
**Основание:** Анализ структуры репозитория, ADR и существующей документации.

---

### 1. ОБЩЕЕ ОПИСАНИЕ

#### 1.1. Назначение
QuickSort — утилита для Windows 10/11, интегрирующаяся в Проводник через контекстное меню. Предоставляет быстрый доступ к избранным папкам, позволяя перемещать/копировать файлы одним кликом. Включает графический редактор для управления папками, историю операций с возможностью отката и системный трей.

#### 1.2. Ключевые сценарии
- **Быстрое перемещение** – ПКМ на файле → «QuickSort» → выбор папки из подменю → перемещение.
- **Расширенный выбор** – ПКМ → «Другие папки…» → открывается окно выбора со списком всех папок.
- **Управление папками** – GUI-редактор (добавление, удаление, переименование, сортировка, избранное).
- **История и откат** – просмотр всех операций и отмена последней (включая восстановление удалённых).
- **Системный трей** – иконка в трее для быстрого открытия редактора и выхода.

### 2. АРХИТЕКТУРА

#### 2.1. Принципы (ADR)
- **Clean Architecture + DDD** – бизнес-логика изолирована от внешних систем (ADR-001, ADR-003).
- **Dependency Rule** – зависимости направлены внутрь, к Domain (ADR-002).
- **Domain Events** – все изменения состояния порождают события (ADR-005).
- **Стабильные порты** – интерфейсы между слоями определены и стабильны (ADR-006).
- **Операции как первичные сущности** – каждая операция (Move, Copy, Delete, Rename) представлена отдельным Use Case (ADR-004).

#### 2.3. Межпроцессное взаимодействие (IPC)
- **Named Pipe** – используется для связи между `context-menu-dll` (загруженной в explorer.exe) и основным процессом Tauri.
- **Контракт** – описан в крейте `quicksort-ipc-contract` (сериализация/десериализация сообщений).
- **Сервер** – реализован в `src-tauri/src/ipc/server.rs`, принимает команды от DLL.
- **Клиент** – реализован в `context-menu-dll/src/pipe_client/`, отправляет запросы к серверу.
- **Протокол** – простой текстовый/бинарный фрейминг (см. `src-tauri/src/ipc/framing.rs`).

#### 2.4. Коммуникация Tauri ↔ Фронтенд
- Используется `@tauri-apps/api/core` – вызовы команд через `invoke()`.
- Команды определены в `src-tauri/src/commands/` (folders, move_cmd, settings).

### 3. КЛЮЧЕВЫЕ МОДУЛИ И ИХ ОТВЕТСТВЕННОСТЬ

#### 3.1. Domain (`quicksort-domain`)
- **Сущности** – `Folder`, `Operation` (содержит тип, источник, цель, статус, временную метку).
- **Value Objects** – `FolderId`, `OperationId`, `WindowsPath` (обёртка над `PathBuf`).
- **События** – `FolderAdded`, `FolderRemoved`, `OperationCompleted`, `OperationUndone`.
- **Доменные сервисы** – вычисление обратной операции, валидация путей.

#### 3.2. Application (`quicksort-application`)
- **Inbound порты** (интерфейсы Use Case):
  - `ExecuteOperation` – выполнение Move/Copy/Delete/Rename.
  - `UndoOperation` – откат последней операции.
  - `GetFolders` – получение списка папок с фильтрацией/сортировкой.
  - `ManageFolders` – CRUD для папок.
- **Outbound порты** (интерфейсы инфраструктуры):
  - `ConfigurationRepository` – чтение/запись `folders.json`.
  - `OperationRepository` – чтение/запись `operations.json`, получение последней операции.
  - `FileSystem` – асинхронные операции с файлами (rename, copy, delete, exists).
  - `IdGenerator` – генерация UUID v7.
  - `Clock` – получение текущего времени.
  - `ConflictResolver` – разрешение конфликтов имён (Skip/Overwrite/AutoRename/Ask).
- **DTO** – `OperationCommand`, `OperationResult`, `PipeCommand` (для IPC).
- **Фасад** (`Facade`) – единая точка входа для Tauri-команд, объединяет все Use Case.

#### 3.3. Infrastructure (`quicksort-infrastructure`)
- **Реализации репозиториев**:
  - `JsonConfigurationRepository` – хранит папки в `%APPDATA%\QuickSort\folders.json`.
  - `JsonOperationRepository` – хранит историю операций в `operations.json` (ограничение 500 записей).
  - `InMemoryOperationRepository` – для тестов.
- **Файловая система** – `StdFileSystem` (обёртка над `tokio::fs`).
- **Генератор ID** – `UuidGenerator` (UUID v7).
- **Часы** – `SystemClock`.
- **Разрешитель конфликтов** – `DefaultConflictResolver` (стратегии переименования).

#### 3.4. Context Menu DLL (`context-menu-dll`)
- **COM-сервер** – реализует интерфейсы `IShellExtInit` и `IContextMenu`.
- **Регистрация** – через `.reg` файл (`activation.reg`) или инсталлятор.
- **Логирование** – в `%TEMP%\quicksort_dll.log`.
- **IPC клиент** – подключается к Named Pipe серверу Tauri, отправляет запрос на выполнение операции или открытие окна выбора.

#### 3.5. GUI (Tauri + React)
- **Бэкенд (Rust)** – `src-tauri/`:
  - Точка входа (`main.rs`) – инициализация трея, CLI, IPC-сервера, регистрация команд.
  - Команды – обёртки над фасадом.
  - IPC-сервер – слушает Named Pipe, десериализует команды, вызывает фасад, возвращает результат.
  - Управление состоянием – `AppState` содержит фасад и путь к исполняемому файлу.
- **Фронтенд (React)** – `src/`:
  - Страницы: `EditorPage` (управление папками), `LogPage` (история), `SelectorPage` (выбор папки), `SettingsPage`, `AboutPage`.
  - Компоненты: `FolderList`, `AddFolderButton`, `ApplyButton`, `StatusIndicator`.
  - Использует Ant Design 5, тёмная тема по умолчанию.
- **Сборка** – через Vite, Tauri CLI (`npm run tauri build`).

### 8. РАЗВЁРТЫВАНИЕ

#### 8.1. Режимы запуска
- **GUI** – `quicksort.exe` (открывает редактор).
- **CLI Move** – `quicksort.exe move <target_dir> <file_path>` (используется DLL).
- **CLI Select** – `quicksort.exe select-folder <file_path>` (открывает окно выбора).
- **CLI Undo** – `quicksort.exe undo` (откат).
- **CLI Admin** – `quicksort-admin.exe install/uninstall` (регистрация COM).

#### 8.2. Установка
- Распаковать архив в `%ProgramFiles%\QuickSort` (или пользовательскую папку).
- При первом запуске создаётся `%APPDATA%\QuickSort\` и файлы конфигурации.
- Регистрация COM-сервера через кнопку в GUI или `quicksort-admin.exe install`.

#### 8.3. Удаление
- Удалить записи из реестра (через GUI или `quicksort-admin.exe uninstall`).
- Удалить папку `%APPDATA%\QuickSort\`.
- Удалить файлы приложения.

### 12. РИСКИ И МИТИГАЦИИ

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Ошибки в COM-сервере приводят к падению Проводника | Средняя | Критическое | Логирование всех операций, изолированное тестирование, использование `catch_unwind` |
| Нестабильность Named Pipe при высокой нагрузке | Низкая | Высокое | Использование таймаутов, повторных попыток, мониторинг |
| Конфликты при откате (файл уже существует) | Высокая | Среднее | Реализовать стратегии разрешения (ConflictResolver) |
| Производительность при большом количестве папок (>1000) | Низкая | Среднее | Виртуализация списка, индексация, кэширование |
| Несовместимость с Windows 10 LTSC или старыми билдами | Низкая | Среднее | Тестирование на целевых версиях, использование стабильных API |
| Утечка памяти в DLL | Низкая | Высокое | Использование RAII, периодическое тестирование с Valgrind/DrMemory |

- `/review` – ревью кода

#### 13.3. Требования к изменениям
- **Соблюдение Dependency Rule:** не нарушать зависимости (Domain не зависит от Application и т.д.)
- **Документирование:** все публичные функции должны иметь `///` комментарии.
- **Тесты:** для каждого Use Case – минимум 3 сценария (успех, ошибка, крайний случай).
- **Логирование:** использовать `tracing::info!`, `tracing::error!` для ключевых действий.
- **Обработка ошибок:** использовать `anyhow::Result` для ошибок на уровне Application, конкретные ошибки в Domain.
- **Форматирование:** `cargo fmt`, `npx prettier` – обязательны.

- **SPECIFICATION.md** – основной документ (этот файл).
- **ADR** – в `docs/adr/` – архитектурные решения.
- **Стандарты** – в `docs/standards/` (rust-style.md, testing.md, error-handling.md).
- **Модели** – в `docs/architecture/` (domain-models.md, glossary.md).
